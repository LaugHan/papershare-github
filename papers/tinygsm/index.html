TinyGSM: achieving >80% on GSM8k with small language models

Bingbin Liu et al. | CMU + Microsoft Research | arXiv:2312.09241 | 2023
摘要（原文翻译）
小规模模型提供了各种计算优势，但规模对问题解决能力的重要程度仍然是一个开放性问题。具体对于小学数学问题的求解，此前达到80%准确率的最小模型尺寸仍然是34B参数。我们的工作研究了高质量数据集可能是小语言模型获得数学推理能力的关键。我们提出了TinyGSM，一个包含1230万个小学数学问题的合成数据集，每个问题配有Python解决方案，全部由GPT-3.5生成。在TinyGSM上微调后，我们发现一个1.3B生成模型和1.3B验证模型的组合可以达到81.5%的准确率，超越了比它大几个数量级的现有模型。这也与GPT-3.5"教师"模型（77.4%）的性能相媲美，而我们模型的训练数据正是由GPT-3.5生成的。我们的方法简单明了，有两个关键组成部分：1）高质量数据集TinyGSM，2）使用验证器从多个候选生成中选择最终输出。

1. 方法动机

a) 驱动力

作者想探索：小规模模型（SLM）能否在数学推理任务上与大规模模型竞争？具体目标是：在GSM8K benchmark上突破80%准确率，同时保持模型尺寸友好。

b) 现存痛点

  • 此前达到80% GSM8K准确率的模型都是34B参数以上，推理成本高昂
  • 小模型（如1-2B参数）在数学推理上表现不佳，能力差距与模型规模强相关
  • GSM8K训练集只有7473道题目，数据量太小不足以训练语言模型
  • 学生模型（SLM）性能通常远落后于教师模型（LLM），存在信息论瓶颈

c) 研究假设

核心假设：高质量合成数据可能是小语言模型获得数学推理能力的关键。通过prompt工程让GPT-3.5生成大量高质量数学问题，可以弥补模型规模的不足。

2. 方法设计（重点）

a) Pipeline 流程

整体架构：生成器 + 验证器的双模型系统

Step 1: 数据生成（TinyGSM Dataset）

  • 输入：GSM8K训练集的7473道原始题目
  • 处理：用GPT-3.5-turbo对每道题生成10个变体（相似问题+Python解答）
  • 输出：1230万个数学问题-答案对（约18亿tokens）

Step 2: 生成器微调（Generator Fine-tuning）

  • 输入：TinyGSM合成数据
  • 模型：基于Phi-1.5（1.3B参数）
  • 训练：FP16、Adam优化器、线性warmup、最大学习率1e-4、batch size 1024、最多20k步
  • 输出：可生成Python代码解答数学问题的生成模型

Step 3: 验证器训练（Verifier Training）

  • 关键发现：验证器大小缩放比生成器大小缩放更有效
    • 生成器从125M→1.3B：性能提升5.1%
    • 验证器从125M→1.3B：性能提升7.2%
  • 训练数据：GSM8K原始训练集（7.4k题目）
  • 作用：对多个候选答案打分，选择最高分作为最终输出

Step 4: 测试时推理（Test-time Inference）

  • 过程：生成器生成48个候选答案 → 验证器打分 → 选择最高分
  • 技巧：include irrelevant context（无关上下文）增强鲁棒性

b) 数据生成Prompt

核心Prompt模板：
"Consider the following grade-school math problem: {question}
Generate 10 different math problems similar to this math problem.
- Make sure each question uses diverse NLP and includes multiple logical steps.
- After each generated problem, write down a detailed and complete Python program..."

3. 与其他方法对比

a) 本质差异

TinyGSM不仅生成问题，还生成配套的Python代码解答，利用Python解释器规避LLM的数值计算局限。

b) 方法对比表

方法 模型规模 GSM8K准确率 改进点
TinyGSM (本文) 1.3B+1.3B 81.5% 合成数据+验证器

WizardMath (LLaMA-2 70B) 70B 81.6% RLEIF强化学习
MAmmoTH (Code-Llama 34B) 34B 72.7% 代码微调
GPT-3.5 (教师) - 77.4% -

c) 创新点

  • 数据规模：1230万道题目，远超MetaMath的4万道
  • 验证器设计：揭示验证器缩放比生成器缩放更有效
  • 无需重训练：验证器只需7.4k标注数据即可训练

4. 实验表现与优势

a) 关键结果

  • 1.3B模型达到81.5%准确率，超越34B+模型
  • 与GPT-3.5教师（77.4%）性能持平
  • Phi-1.5微调前44.6% → 微调后68.2% → 加验证器后81.5%
  • 最小的125M模型也能达到63.1%（无验证器）和68.9%（有验证器）

b) 优势场景

  • 小学数学应用题（GSM8K）
  • 需要Python代码执行验证的任务
  • 计算资源受限但需要数学推理能力的场景

c) 局限性

  • 泛化限制：TinyGSM针对GSM8K风格设计，对更高阶数学（MATH数据集）效果有限
  • 验证器依赖：需要可执行的Python代码，非代码任务不适用
  • 教师模型依赖：数据质量受限于GPT-3.5能力
  • 无正确答案验证：合成数据未检查正确性（作者发现一致性过滤会移除难题）

5. 学习与应用

a) 开源情况

代码和数据已开源（GitHub + HuggingFace）

b) 实现细节

  • 数据生成成本：约$3600（GPT-3.5商业定价）
  • 训练步数：最多20k步
  • 采样温度：用于增强多样性
  • 候选数量：测试时生成48个候选答案

c) 迁移能力

可迁移到其他需要"可执行验证"的领域：

  • 其他编程语言的代码生成
  • 形式化数学证明（如果有验证器）
  • 任何有明确验证函数的推理任务

6. 总结

a) 一句话概括（≤20字）

高质量合成数据+验证器选择，让1.3B小模型在数学推理上比肩GPT-3.5。

b) 速记版Pipeline（3-5步）

  1. 收集种子：准备少量真实数学题（如GSM8K训练集）
  2. 大模型生成：让GPT-3.5生成大量相似问题+Python解答
  3. 过滤清洗：删除过短、无数字、不可执行的题目
  4. 微调生成器：在合成数据上微调小模型
  5. 训练验证器：用少量真实数据训练验证器模型
  6. 测试选择：生成多个候选，验证器打分选最优
